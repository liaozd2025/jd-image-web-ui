# 服务器部署成为唯一产品形态

本仓库从原版单人本机工作台单向改造为部门服务器产品。系统统一部署在服务器上，所有用户通过浏览器登录访问；不再维护桌面 App、便携包、本机工作流、Codex / ChatGPT OAuth 或面向用户的生图 CLI。保留的 CLI 仅用于管理员初始化、账号恢复、备份恢复和数据维护。这样可以用一套身份、权限、数据归属和审计模型覆盖整个产品，避免继续承担双模式带来的安全分支与维护成本。

## Considered Options

- 同仓库同时维护本机模式和服务器模式：有利于持续合并上游，但认证、存储、队列、供应商和测试都要维护两套语义。
- 服务器主体加用户电脑本地桥接：可以复用每个人的 Codex OAuth，但需要安装本地代理并解决身份绑定、通信安全和版本管理。
- 纯服务器、浏览器访问：放弃本机 OAuth 和桌面兼容，换取清晰的多用户边界与统一交付方式。本仓库选择此方案。

## Consequences

- 第一个可部署版本必须同时完成登录认证、用户数据隔离、受保护的文件访问、个人与共享资产、个人与部门供应商、额度与存储配额、公平队列和审计；不得交付“只有登录但数据仍共享”的过渡版本。
- 服务器只调用管理员白名单中的 OpenAI-compatible API。个人 API Key 和部门 API Key 加密保存，完整密钥不返回浏览器，也不进入任务、日志或审计记录。
- 关系数据使用 PostgreSQL，图片与其他大文件使用服务器文件数据卷。数据库通过 psycopg 和版本化 SQL 迁移维护，不使用 ORM。
- Web 与 Worker 分进程运行，任务状态保存在 PostgreSQL。等待任务可在重启后续排，执行中任务标记为中断且不自动重试。
- 标准交付为 Linux Docker Compose：HTTP 反向代理、Web、Worker 和 PostgreSQL。部署目标是公司内网，首期不启用 TLS，但应用容器端口不直接暴露。
- 首次服务器部署使用全新数据目录，不自动迁移旧版图库、历史任务、供应商配置、API Key 或 OAuth 数据。
- 原版桌面打包、托盘启动器、便携包、更新器、本机 OAuth 客户端和用户生图 CLI 可逐步删除；上游功能以后按需选择性同步。
