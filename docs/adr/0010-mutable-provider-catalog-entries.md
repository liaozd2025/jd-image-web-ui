# 供应商目录条目原位修改并使用任务快照保留历史

管理员编辑现有供应商目录条目时，系统原位修改该条目，不创建新的供应商或下一
版本。`provider_version_id`、`version_number` 和启停状态保持不变；供应商标识
继续不可编辑。名称、Base URL、API 模式、模型配置和部门凭据可以更新。

历史任务不依赖当前目录字段解释当时的模型行为，而是使用任务提交时保存的模型、
能力档案、协议和非敏感请求快照。API Key 仍只保存加密值，不进入任务快照、响应
或审计详情。

## Considered Options

- 每次编辑创建不可变版本：历史边界清晰，但会把一次普通修改表达成新的供应商，
  停用条目编辑还需要复制凭据，目录容易堆积重复条目。
- 原位修改目录条目，任务自行保存执行快照：管理员看到的始终是当前供应商，
  同时历史任务仍保留当时的模型和能力信息。本仓库选择此方案。

## Consequences

- 编辑已有条目只更新原记录，不能增加版本号，也不能自动改变启停状态。
- `POST /api/admin/provider-catalog` 只创建标识唯一的新条目；已有条目只允许通过
  `PATCH /api/api-settings` 原位更新，PATCH 不承担创建语义。
- Base URL、API 模式、Model ID、协议或凭据变化后，相关模型验证状态失效。
- 生图模型尽量保留原 `generation_model_id`；被任务引用的模型只能停用，不能
  物理删除。
- 产品中的“删除供应商”是不可恢复入口的软删除：条目立即停用并从目录、工作台
  和运行时查询中隐藏，历史任务、外键和审计记录继续保留。
- 目录至少保留一个未删除供应商；删除最后一个条目必须被拒绝。
- 数据库中的 `provider_catalog_versions` 和相关版本字段作为兼容名称保留，新代码
  不再把它们解释为“每次编辑都创建一条不可变版本记录”。
