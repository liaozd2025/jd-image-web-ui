# 生图模型使用系统内置能力档案

系统为已支持的生图模型维护内置模型能力档案，例如 Seedream 5.0 Lite、
Seedream 5.0 Pro。部门管理员或个人供应商所有者添加生图模型时填写显示名称和
供应商控制台中的 Model ID，并选择对应的内置能力档案；配置者不能自行勾选底层
能力或填写自由 JSON 参数。

系统尚未识别的 Model ID 只能使用保守的基础能力档案。新的高级能力必须随系统
支持和测试一起加入能力档案，不能仅通过修改管理配置直接开放。

个人供应商用户可以自由填写 Model ID，但仍必须为每个模型选择系统内置能力档案；
未知模型使用“通用基础”档案。个人用户不能自行声明尺寸、参考图数量、流式输出
等能力，也不能填写自由 JSON 能力参数。这一限制不属于管理员审核。

部门供应商的新模型先保存为未验证，管理员可以执行一次最小规格的真实生图测试。
验证用于诊断供应商配置并保留模型、时间、供应商请求编号和成功或失败结果，但不再
阻止已启用的团队模型进入图片工作区或提交任务。Model ID、接口地址或部门供应商
凭据变化后，相关模型回到未验证状态并可重新测试。验证图片不进入个人图库或共享
图库，也不记录 API Key。个人供应商不执行管理员验证，其自主管理边界见 ADR 0009。

## Considered Options

- 根据 Model ID 在前端硬编码：开发成本低，但前后端容易产生不同判断，供应商
  调整命名后也难以维护。
- 允许管理员自由配置能力和任意 JSON：适配新模型快，但容易声明供应商实际不
  支持的组合，也可能把未经校验的参数直接送往外部接口。
- 运行时自动查询供应商能力：理想情况下最及时，但火山方舟当前没有提供满足
  本产品需要的完整、稳定模型能力发现契约。
- 系统内置、前后端共享的能力档案：上线新能力需要更新系统，但行为可测试、
  可解释，并能保持界面和服务端校验一致。本仓库选择此方案。

## Consequences

- 生图模型配置必须引用一个模型能力档案，而不是保存散落的能力开关。
- 供应商设置使用结构化模型列表，不使用逗号分隔的模型文本框。每行包含显示名称、
  Model ID、能力模板、默认模型和启用状态；同一供应商只能有一个默认模型。
- Seedream 5.0 Lite 与 Seedream 5.0 Pro 使用不同能力档案，即使共享同一供应商
  凭据与接口地址。
- 已被任务引用的模型只能停用，不能物理删除；历史任务继续保留原模型信息。
- 未知模型可以完成基础生成，但不会显示连续组图、精准编辑、流式返回等高级入口。
- 个人供应商可以使用任意 Model ID，但该自由度不改变能力档案和服务端参数校验。
- 部门供应商中已启用的模型均可出现在图片工作区的模型选择器中；未验证或验证失败
  状态继续在系统设置中展示，供管理员诊断，但不作为团队模型的使用门禁。
- 部门供应商的 Model ID、接口地址或凭据变更必须使相关验证状态失效；验证调用
  使用模型能力档案允许的最小规格，避免产生不必要费用。
- 验证图片在确认接口返回有效图片后立即清理，不进入用户资产、任务历史或图库。
- 新增或修改能力档案必须同时补充请求映射、服务端校验和界面行为测试。
- 任务记录能力档案标识或等价版本快照，避免档案更新后改变历史任务含义。
- 首个交付阶段包括多模型配置与选择、任务快照、能力约束与动态表单、Prompt 优化、
  Seed、水印、PNG/JPEG，以及历史任务重试和服务端校验。
- 第二阶段依次交付 Pro 画布精准编辑、Lite 连续组图、Lite 流式返回；这些能力
  不得在首个交付阶段以未完成或不可验证的入口提前出现。
